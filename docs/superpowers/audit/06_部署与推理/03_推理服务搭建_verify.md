## 二、测试用例

### 2.1 API 功能测试

#### TC-001: 健康检查端点
| 项目 | 内容 |
|------|------|
| **测试目标** | 验证 API 服务健康检查端点可用 |
| **前置条件** | 服务已启动 |
| **测试步骤** | 1. 发送 GET 请求到 `/health` 或 `/`<br>2. 检查响应状态码<br>3. 检查响应体格式 |
| **预期结果** | 状态码 200，响应体包含 `{"status": "healthy"}` 或等效字段 |
| **验证方法** | 自动化测试 + 手动 curl 验证 |
| **通过标准** | 连续 3 次请求均返回正确响应 |

#### TC-002: 图片 URL 输入推理
| 项目 | 内容 |
|------|------|
| **测试目标** | 验证通过图片 URL 进行推理 |
| **前置条件** | 服务已启动，测试图片 URL 可访问 |
| **测试数据** | 准备 3 张不同类别的测试图片 URL（PNG、JPG 格式） |
| **测试步骤** | 1. POST 请求到 `/predict`，body: `{"image_url": "<url>"}`<br>2. 检查响应状态码<br>3. 验证响应体包含 `class_name` 和 `confidence` |
| **预期结果** | 状态码 200，响应格式: `{"class_name": "xxx", "confidence": 0.xx}` |
| **验证方法** | 自动化测试 |
| **通过标准** | 3 张图片均返回正确格式响应，confidence ∈ [0, 1] |

#### TC-003: Base64 图片输入推理
| 项目 | 内容 |
|------|------|
| **测试目标** | 验证通过 Base64 编码图片进行推理 |
| **前置条件** | 服务已启动，准备 Base64 编码的测试图片 |
| **测试数据** | 1 张标准测试图片的 Base64 编码字符串 |
| **测试步骤** | 1. POST 请求到 `/predict`，body: `{"image_base64": "<base64_str>"}`<br>2. 检查响应状态码和格式 |
| **预期结果** | 状态码 200，返回分类结果和置信度 |
| **验证方法** | 自动化测试 |
| **通过标准** | 返回有效分类结果，与 TC-002 同图片结果一致 |

#### TC-004: 文件上传推理
| 项目 | 内容 |
|------|------|
| **测试目标** | 验证通过文件上传进行推理 |
| **前置条件** | 服务已启动，准备测试图片文件 |
| **测试数据** | PNG、JPG 各 1 张 |
| **测试步骤** | 1. POST 请求到 `/predict`，multipart/form-data 上传文件<br>2. 检查响应状态码和格式 |
| **预期结果** | 状态码 200，返回分类结果和置信度 |
| **验证方法** | 自动化测试 |
| **通过标准** | 两种格式均正常处理并返回结果 |

#### TC-005: 输出格式验证
| 项目 | 内容 |
|------|------|
| **测试目标** | 验证输出格式符合规范 |
| **前置条件** | 服务已启动 |
| **测试步骤** | 1. 发送有效推理请求<br>2. 验证响应字段完整性<br>3. 验证数据类型正确性 |
| **预期结果** | 响应必须包含:<br>- `class_name`: string<br>- `confidence`: float ∈ [0, 1]<br>- 可选: `class_id`: int, `probabilities`: array |
| **验证方法** | 自动化测试（JSON Schema 验证） |
| **通过标准** | 所有必填字段存在且类型正确 |

#### TC-006: 置信度范围验证
| 项目 | 内容 |
|------|------|
| **测试目标** | 验证置信度值在有效范围内 |
| **前置条件** | 服务已启动 |
| **测试步骤** | 1. 发送 10 次不同图片的推理请求<br>2. 记录所有返回的 confidence 值 |

### 5.2 CI/CD 集成建议
```yaml
# .github/workflows/test.yml 示例
stages:
  - lint
  - unit-test
  - docker-build
  - integration-test

jobs:
  unit-test:
    script: pytest tests/test_api_*.py -v

  docker-build:
    script: docker build -t inference-service:test .

  integration-test:
    script: |
      docker run -d -p 8000:8000 inference-service:test
      pytest tests/test_docker.py -v
```

### 5.3 自动化命令示例
```bash
# 运行所有自动化测试
```

## 六、风险点

### 6.1 技术风险
| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|----------|
| 模型加载内存不足 | 服务启动失败 | 中 | 限制模型大小，使用模型量化 |
| GPU 不可用导致推理失败 | 生产环境不可用 | 中 | 提供 CPU fallback 模式 |
| 图片下载超时 | URL 输入失败 | 中 | 设置合理超时时间 (30s) |
| 大文件上传 OOM | 服务崩溃 | 低 | 限制上传文件大小 (10MB) |
| 并发请求过多 | 服务响应慢或崩溃 | 中 | 实现请求队列和限流 |

### 6.2 测试风险
| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|----------|
| 测试图片 URL 失效 | 测试失败 | 中 | 使用本地测试图片或稳定图床 |
| Docker 环境差异 | 构建失败 | 低 | 使用多阶段构建，明确基础镜像版本 |
| 网络环境不稳定 | 性能测试结果不准确 | 低 | 多次运行取平均值 |

### 6.3 依赖风险
| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|----------|
| FastAPI 版本不兼容 | API 行为异常 | 低 | 锁定依赖版本 (requirements.txt) |
| PyTorch 版本问题 | 推理结果不一致 | 中 | 使用与训练相同版本 |
| Python 版本差异 | 依赖安装失败 | 低 | 明确 Python 版本要求 (≥3.8) |
